EdgeVector

Personal AI Knowledge Infrastructure
A Technical Whitepaper
Edge Vector Foundation, 2026

Artificial intelligence systems increasingly assist users in reasoning over complex information. However, current AI tools depend on centralized cloud infrastructure that requires users to surrender private data to remote servers. This architecture is fundamentally incompatible with privacy, data sovereignty, and the nature of personal knowledge.

This paper introduces EdgeVector, a new architecture for personal AI knowledge infrastructure that runs entirely on local edge devices. The reference implementation, FoldDB, is a database system that converts a user's local storage into an AI-queryable knowledge corpus — without any data leaving the machine.

We describe the system architecture, ingestion pipeline, query engine, encryption model, and performance characteristics. We argue that the convergence of capable local language models, high-capacity NVMe storage, and growing personal data volumes creates a unique opportunity to shift AI infrastructure from the cloud to the edge.

1. Introduction

1.1 The Problem

Modern knowledge work produces a vast and growing archive of personal data: source code, documents, emails, notes, research papers, configuration files, and more. These artifacts collectively represent a user's intellectual corpus — the accumulated output of their professional and creative work.

AI assistants promise to make this corpus searchable and useful. In practice, however, every major AI platform requires users to upload their data to centralized cloud systems. This introduces several critical problems:

  1. Privacy exposure. Sensitive data — proprietary code, confidential documents, personal communications — must be transmitted to and stored on third-party servers.
  2. Fragmented access. AI tools operate on individual documents, repositories, or conversations. No existing system provides unified semantic access across a user's entire data corpus.
  3. No local retrieval infrastructure. File systems organize data by location, not by meaning. Local AI models have no structured index to query against.
  4. Vendor lock-in. Cloud-based AI knowledge systems bind users to specific platforms, making data portability difficult or impossible.

The fundamental tension is clear: AI needs structured access to data, but users need sovereignty over that data. These goals are treated as mutually exclusive by current architectures.

1.2 Our Thesis

EdgeVector proposes that this tension is resolvable. The key insight is:

AI knowledge infrastructure does not need to be centralized. It can run locally, on the same machine where the data already lives.

Three converging trends make this feasible now:

  • Local large language models. Models in the 2B–13B parameter range run efficiently on consumer hardware with acceptable latency and quality for retrieval-augmented generation (RAG) tasks.
  • High-capacity NVMe storage. Multi-terabyte NVMe drives are commodity hardware, capable of sustaining the I/O throughput required for large-scale file indexing.
  • Personal data explosion. The volume of data generated per knowledge worker continues to grow, increasing both the need for and the value of semantic search.

2. System Architecture

FoldDB is a local AI knowledge database. Unlike traditional databases that store application data, FoldDB indexes existing files on the user's filesystem and constructs semantic retrieval layers over them. The system is designed to be used by local AI models as a knowledge backend.

The architecture consists of six core subsystems:

  1. File Ingestion Engine — scans and parses files
  2. Metadata Extractor — derives structured metadata from content
  3. Vector Store — stores embeddings for semantic search
  4. Delta Scanner — detects and processes file changes
  5. Query Engine — combines multiple retrieval strategies
  6. Encryption Layer — protects content and keys at rest

2.2 Design Principles

  • Zero exfiltration. No data, metadata, telemetry, or analytics leave the local machine under any circumstance.
  • Index, don't copy. FoldDB creates retrieval layers over existing files. It does not duplicate or relocate user data.
  • Format-agnostic. The system supports a broad whitelist of file types through pluggable parsers.
  • Background-first. Indexing and re-indexing happen as background processes that do not block user interaction.

3. File Ingestion Pipeline

3.1 Supported Formats

File types outside this whitelist are ignored. The whitelist is user-configurable and extensible.

3.2 Parsing Strategy

Each file type has a dedicated parser that extracts two outputs:

  1. Text content — the raw textual content of the file, normalized to UTF-8.
  2. Structured metadata — format-specific attributes extracted during parsing.

For source code files, the parser additionally extracts:

  • Function and method names
  • Class and type definitions
  • Comments and docstrings
  • Import/dependency declarations
  • File structure (module hierarchy)

Parsing is implemented as a pipeline of composable stages:

File → Format Detection → Parser Selection → Content Extraction
                                               → Metadata Extraction
                                               → Chunk Generation
                                               → Embedding Generation

3.3 Chunking

Documents are split into overlapping chunks for embedding. The chunking strategy is format-aware:

  • Code: chunks align to function/class boundaries where possible.
  • Prose: chunks split on paragraph or section boundaries.
  • Data: chunks split on record boundaries (rows, objects).

Chunk size targets 512 tokens with 64-token overlap, configurable per format.

4. Delta Scanner

4.1 Purpose

The delta scanner is a background daemon that keeps the index synchronized with the filesystem. Rather than re-indexing the entire drive on each pass, it detects only the files that have changed and processes them incrementally.

4.2 Algorithm

1. Walk the target directory tree
2. For each file:
   a. Compute content hash (BLAKE3)
   b. Compare against stored hash in metadata store
   c. If hash differs or file is new:
      - Re-parse content
      - Update metadata
      - Regenerate embeddings
      - Update vector index
   d. If file is deleted:
      - Remove from all indexes
3. Record scan timestamp
4. Sleep until next scan interval

4.3 Performance Characteristics

  • Hashing: BLAKE3 processes data at memory bandwidth speeds (>10 GB/s on modern hardware), making hash comparison negligible in cost relative to I/O.
  • Incremental updates: Only changed files trigger the full parse-embed pipeline. A typical scan of 500,000 files with <1% change rate completes in seconds.
  • Filesystem events: On supported platforms, the scanner optionally subscribes to filesystem notification APIs (FSEvents on macOS, inotify on Linux) to detect changes without polling.

5.2 Storage

Metadata is stored in a local embedded database (Sled or SQLite). The metadata store supports full-text search over extracted text fields, filtered queries by file type, date range, tags, or symbols, and fast lookup by file path or content hash.

6. Vector Index

6.1 Embedding Generation

Each chunk is embedded using a local embedding model. The default model is a quantized sentence-transformer (e.g., all-MiniLM-L6-v2 or bge-small-en-v1.5) running on CPU or GPU. Embedding dimensions: 384 (configurable based on model choice).

6.2 Index Structure

The vector index uses HNSW (Hierarchical Navigable Small World) graphs for approximate nearest neighbor search. HNSW provides sub-millisecond query latency at million-scale vector counts, low memory overhead with memory-mapped storage, and incremental insertion without full rebuilds.

6.3 Storage Layout

├── vector_index/

│   └── vectors.bin          # Raw embedding vectors
├── document_store/
│   ├── chunks/              # Serialized chunk content

└── encryption_keys/

    └── keyring.db           # Per-file key derivations

7. Query Engine

7.1 Multi-Strategy Retrieval

The query engine combines three retrieval strategies and merges their results:

  1. Semantic search. The user's query is embedded and compared against the vector index using cosine similarity. Returns the top-k most semantically relevant chunks.
  2. Keyword search. BM25 or full-text search over the metadata store. Catches exact matches that semantic search may rank lower.
  3. Metadata filtering. Structured filters narrow results by file type, date range, language, symbol name, or tags.

Results from all three strategies are merged using reciprocal rank fusion (RRF), which produces a unified ranking without requiring score normalization across heterogeneous retrieval methods.

7.2 Query API

FoldDB exposes a local HTTP API and an MCP (Model Context Protocol) interface for AI agent integration.

REST API:

POST /v1/search
{
  "query": "distributed encryption design decisions",

MCP Interface: FoldDB registers as an MCP tool provider, exposing search, get_document, and list_files tools that any MCP-compatible AI agent can invoke directly.

8. Security Model

8.1 Threat Model

FoldDB assumes the following threat model:

  • Trusted: The local machine and its operating system.
  • Untrusted: All networks, cloud services, and external processes.
  • Goal: Even if the FoldDB data directory is exfiltrated (e.g., stolen laptop), the content must remain unreadable without the user's passphrase.

8.2 Encryption Architecture

  • The master key is derived from the user's passphrase using Argon2id with high memory and iteration parameters.
  • Per-purpose keys are derived from the master key using HKDF.
  • The master key is never stored in plaintext. It is held in memory only while the application is running and the user has authenticated.
  • A public/private keypair (Ed25519) is generated at first launch for future node-to-node authentication in distributed scenarios.

8.4 Zero-Exfiltration Guarantee

FoldDB enforces a strict no-network policy:

  • The application makes zero outbound network connections during normal operation.
  • The local API binds exclusively to 127.0.0.1.
  • No telemetry, crash reports, or update checks are transmitted.
  • Network access is only used if the user explicitly enables optional cloud sync (P1 feature).

9. AI Integration

9.1 Local LLM

FoldDB ships with a bundled 2B-parameter language model for out-of-the-box functionality. Users can configure any local model that exposes an OpenAI-compatible API (Ollama, llama.cpp, LM Studio, etc.).

9.2 Retrieval-Augmented Generation (RAG)

The standard query flow follows a RAG pattern:

  1. User submits natural language query
  2. Query engine retrieves relevant chunks (Section 7)
  3. Retrieved chunks are injected into the LLM prompt as context
  4. LLM generates a grounded response with source citations
  5. Response includes file paths and chunk references for verification

9.3 Agent Tooling

For AI agent frameworks, FoldDB functions as a tool provider:

  • MCP server — compatible with Claude, Cursor, and other MCP-aware agents
  • Function calling — exposes search/retrieve/list as callable tools
  • Streaming — supports streaming chunk retrieval for long-running agent workflows

This allows any local AI agent to treat the user's entire filesystem as a queryable knowledge base.

10.2 Optimizations

  • Parallel ingestion. File parsing and embedding generation are parallelized across available CPU cores.
  • Memory-mapped I/O. The vector index and metadata store use memory-mapped files, allowing the OS to manage caching efficiently.
  • Lazy embedding. Embeddings are generated on-demand for newly added files and cached persistently. Unchanged files are never re-embedded.
  • Tiered storage. Hot metadata (frequently queried) remains in memory; cold data is accessed via memory-mapped disk.

11. Applications

11.1 Near-Term (v1)

  • Personal knowledge search. Ask natural language questions across all local files.
  • Codebase understanding. Query function definitions, design patterns, and architectural decisions across repositories.
  • Document recall. Find documents by content rather than filename or folder.
  • Research synthesis. Surface connections between papers, notes, and code across projects.

11.2 Medium-Term (v2)

  • Meeting preparation. Before a meeting, query FoldDB for all relevant context — past notes, related code changes, prior decisions.
  • Cloud sync and resilience. Optional encrypted backup to user-controlled cloud storage.
  • Mobile access. Query your knowledge base from a mobile device via encrypted tunnel to your local machine.

11.3 Long-Term Vision

EdgeVector envisions a future where individuals operate sovereign AI knowledge nodes. Each node indexes its owner's data locally, runs its own AI models, and optionally connects to other nodes in a distributed knowledge network.

In this architecture, collaboration happens through structured queries between nodes, not by centralizing data. Privacy is preserved because each node decides what to share, with whom, and under what conditions.

This is the vision of personal AI infrastructure — the computational and data sovereignty equivalent of personal computing.

13. Organizational Structure

EdgeVector operates as a non-profit foundation (Edge Vector Foundation) with a for-profit subsidiary.

  • The foundation stewards the open-source FoldDB core, maintains the protocol specifications, and governs the ecosystem.
  • The subsidiary builds commercial products on top of the open-source platform: managed cloud sync, enterprise features, mobile applications, and premium support.

This structure ensures that the core technology remains open and community-governed while enabling sustainable commercial development.

14. Conclusion

The current paradigm of centralized AI infrastructure forces a false trade-off between AI capability and data privacy. EdgeVector rejects this trade-off.

By running AI knowledge infrastructure locally — indexing files where they already live, embedding content on the user's own hardware, and encrypting everything at rest — FoldDB demonstrates that AI assistants can understand a user's complete data corpus without that data ever leaving the machine.

The technical foundations are sound: local LLMs are capable enough, NVMe storage is fast enough, and HNSW indexes are efficient enough. What has been missing is the system that ties these components together into a coherent, privacy-preserving knowledge platform.

FoldDB is that system. EdgeVector is the architecture. The shift from centralized AI to personal AI infrastructure begins at the edge.
